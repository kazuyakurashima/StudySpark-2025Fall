# StudySpark 進化ロードマップ

> 作成日: 2026-03-14
> 最終更新: 2026-03-14
> ステータス: 検討中（議論ベースの整理）
> レビュー: Codexレビュー反映済み（スキーマ統一、Phase分割、法務ゲート、ENUM比較安全化）

## 目的

StudySparkを「塾内ツール」から「マネタイズ可能なプロダクト」へ進化させるためのアイデア整理と優先順位付け。
全アイデアを「誰のどんな痛みを解消するか」「誰がいくら払うか」の2軸で評価する。

## 本質的なゴール

> 生徒のモチベーションを上げること。結果として成績が上がること。

全ての機能は、この本質に立ち返って優先度を判断する。

## ロードマップの基本方針

```
Phase 1: まず生徒への価値体験を高める（塾内ツールとして）
  1A: 算数の演習問題集（縦切りで入力→採点→マップ→3ロール表示）
  1B: 理科・社会の横展開
  1C: 旧入力の廃止・整理

Phase 2: 認証本番化 + 一般公開基盤の整備（モバイル化と同時）
  → 価値が証明できたら、有料化・一般公開へのステップを踏む
  → 法務→認証→課金の順でゲート通過

Phase 3: スケール・拡張
  → 他塾展開（テナント分離含む）、UX改善、エンゲージメント強化
```

## アイデア一覧と評価

### A. 高優先：ペインが明確 & マネタイズに直結

#### A1. 演習問題集の問題別入力 + 予習シリーズ入力の段階的廃止

| 項目 | 内容 |
|------|------|
| **誰の痛み** | **生徒**: 毎回「何ページやった」の入力が作業化。コメントも形骸化。**指導者**: どの問題ができて/できないかが把握できず、個別指導に活かせない |
| **解決** | 予習シリーズの入力を段階的に廃止し、演習問題集の問題単位で正誤をiPad入力。問題レベル別の到達度が可視化される |
| **マネタイズ** | 「このアプリでなければできない」コア価値。塾への導入提案の根拠。塾単位のライセンス課金の根拠 |
| **実装規模** | 中〜大 |

##### 実装方針の検討結果

**結論: 科目ごとに随時投入（算数→理科→社会の順）**

一括開発（別ブランチで全科目同時）ではなく、1科目ずつ随時投入する方針を採用する。

理由:
1. **算数を最初にやるべき**。既に`question_sets` / `questions` / `answer_sessions` / `student_answers`テーブルと採点ロジック（`lib/math-grading.ts`）が存在しており、演習問題集への拡張が最も自然
2. **科目ごとに問題の性質が違う**。算数は計算・数値回答、理科・社会は選択肢や記述が混在。1科目ずつやれば、実際の問題を見ながら必要十分な設計ができる
3. **マスタデータ整備が最大のボトルネック**。全科目一括だと全問題のデータ入力完了までリリースできない。随時投入なら「算数の第5回分」ができた時点で価値を届けられる

```
Step 1: 算数の演習問題集 → 既存テーブル（question_sets / questions）を拡張
Step 2: 理科の演習問題集 → 算数の経験を基にスキーマ調整
Step 3: 社会の演習問題集 → ほぼ理科と同じ構造で横展開
（国語は問題集構造が異なるため別途検討）
```

##### コース別到達度の設計（min_course方式）

**課題: 現在の予習シリーズ方式の問題点**

現在の`study_content_types`はコースごとに**別レコード**（Aコースの「類題」とBコースの「類題」は別ID）。
そのため、コース変更すると到達度マップの列が分断され、年間を通じた一貫した表示ができない。

```
【現在の問題 — Bコースに変更後の到達度マップ】

         │ 類題(A) │ 基本(A) │ 類題(B) │ 基本(B) │ 練習(B) │
─────────┼────────┼────────┼────────┼────────┼────────┤
第1〜5回  │  92%   │  85%   │        │        │        │  ← Aコース時
第6〜7回  │        │        │  75%   │  70%   │  60%   │  ← Bコース時
→ 列が分断し、年間の流れが把握できない
```

**解決策: min_course方式**

演習問題集では、問題IDをコース横断で共通化し、各問題に「この問題が表示される最低コース」を持たせる。

```
演習問題集 第1回 算数:
  問1（基本）  → min_course = A  → A/B/C/S 全員に表示
  問2（基本）  → min_course = A  → A/B/C/S 全員に表示
  問3（標準）  → min_course = B  → B/C/S に表示
  問4（標準）  → min_course = B  → B/C/S に表示
  問5（発展）  → min_course = C  → C/S に表示
  問6（発展）  → min_course = S  → S のみ
```

**コース変更時の到達度マップの挙動:**

Aコース（第1〜5回）→ Bコース（第6回〜）に変更した場合:

```
【到達度マップ — 常に「現在のコース＝B」の視点で表示】

         │ 問1(基本) │ 問2(基本) │ 問3(標準) │ 問4(標準) │
─────────┼─────────┼─────────┼─────────┼─────────┤
第1回     │    ○     │    ○     │    −     │    −     │  ← Aコース時のデータが活きる
第2回     │    ○     │    ×     │    −     │    −     │  ← 標準問題は未回答（−）
第3回     │    ○     │    ○     │    −     │    −     │
第4回     │    ○     │    ○     │    −     │    −     │
第5回     │    ×     │    ○     │    −     │    −     │
第6回     │    ○     │    ○     │    ○     │    ×     │  ← Bコースで全問入力
第7回     │    ○     │    ○     │    ×     │    ○     │
```

この設計のメリット:
1. **コース変更してもデータが分断されない** — 問題IDがコース横断で共通のため、同じ列に表示され続ける
2. **コースが上がると列が増える** — 新しい難易度の列が追加され、過去回は「−（未回答）」表示。「ここを遡って解いてみよう」という指導にも使える
3. **到達率が自動調整される** — コースが上がると分母（対象問題数）が増えるため、到達率は相対的に下がり「まだ伸びしろがある」ことが可視化される

```
到達率の変化例:
  第1回（Aコース時）: 2問正解 ÷ 2問（問1〜2） = 100%
  第1回（Bコース後）: 2問正解 ÷ 4問（問1〜4） = 50%  ← 分母が増え、伸びしろが見える
```

##### コース別プレビュー機能

到達度マップに「コース切替プレビュー」を設ける。

**目的**: 実際のコース設定を変更せずに、他コースの視点で到達度を確認できるようにする。

```
到達度マップ上部:
  [表示コース: A | B | C | S ]  ← 現在のコースがデフォルト選択
  ※ プレビュー中は「プレビュー表示」ラベルを明示（実コース指標との混同を防ぐ）
```

ユースケース:
- **指導者**: 「この子がAコースだったらどう見えるか」を確認し、コース変更の判断材料にする
- **生徒**: 自分がCコースの場合、Aコースに切り替えて「基本問題はどれだけできているか」を確認
- **保護者**: 子供の到達度を異なるコース基準で把握

**重要**: このプレビューは表示フィルタの変更のみ。`students.course`（実際のコース設定）は変更しない。
実際のコース設定を変えると、Spark画面の入力対象も変わってしまうため、到達度確認と入力設定は分離する必要がある。

**UX上の注意**: プレビューでコースを切り替えると分母が変わり到達率が急変する。「実コースの到達率」と「プレビュー到達率」が混同されないよう、UIで明確に区別する。

実装:
- 到達度マップ表示時のSQLに `WHERE course_rank(min_course) <= course_rank(選択コース)` を適用
- 追加テーブル不要。UIにセレクターを追加するのみ

##### 予習シリーズ入力の廃止方針

**結論: 段階的に廃止**

完全廃止が最終目標だが、一気に消すのではなく移行期間を設ける。

理由:
- 併存すると「入力が作業化している」痛みが解消されないどころか入力先が増えて悪化する
- 予習シリーズの入力データ（ページ数・問題数合計）は分析価値が低く、指導に活かせない

```
Phase A: 演習問題集の入力を追加（算数から）。予習シリーズ入力はそのまま残す
Phase B: 生徒・指導者に「演習問題集の方に入れてね」と案内。利用状況をモニタリング
Phase C: 予習シリーズの新規入力をUI上非表示に。過去データは閲覧可能に残す
Phase D: 完全廃止（非表示のまま放置でもOK）
```

**データ移行の注意**: 既存のAIコーチング（週次分析、日次メッセージ）は`study_logs`を入力源としている。
予習シリーズ廃止後は、演習問題集データ（`answer_sessions` / `student_answers`）もAI分析の入力源に含める必要がある。
移行期間中は両方のデータソースを参照する集計Viewを用意し、分析ロジックの入力源を段階的に移行する。

##### 到達度マップの混在対応

演習問題集の到達度マップを追加すると、予習シリーズの到達度マップと混在する期間が発生するが、**これは意図的な移行期間として設計すれば問題ない**。

混在が問題にならない理由:
1. **データソースが明確に異なる** — 予習シリーズは`study_logs`、演習問題集は`question_sets` / `student_answers`系テーブル。混ざるリスクがない
2. **見たいものが違う** — 予習シリーズは「どの学習内容をやったか」の進捗マップ、演習問題集は「どの問題ができた/できないか」の到達度マップ
3. **段階的廃止と自然に整合** — 予習シリーズの入力が減る → マップのデータが空になる → 非表示にする自然なタイミングが来る

UI上の整理:

```
リフレクト画面のタブ構成（現在）:
  [振り返り] [到達度マップ]

移行期:
  [振り返り] [到達度マップ]
                └─ サブタブ: [演習問題集（デフォルト）] [予習シリーズ]

廃止後:
  [振り返り] [到達度マップ]  ← 演習問題集のみ
```

コンポーネント戦略:
- 最初は演習問題集用のマップを別コンポーネントとして作成
- 2つの実装が安定してから共通部分（ヒートマップグリッド、色ロジック、科目タブ、レスポンシブ、凡例）を抽出
- 過度な抽象化を避けるため、最初から共通コンポーネント化は急がない

##### 付随する廃止対象

- **マスタープリント入力**: 演習問題集入力が実現すれば不要。第5回以降の問題データ整備も省略可能
- **漢字テスト入力**: 授業中に100点まで繰り返すため、アプリに記録する意味がない。フラグで非表示にするだけで対応可

#### A2. モバイル化（PWA）+ 一般公開基盤

| 項目 | 内容 |
|------|------|
| **誰の痛み** | **生徒・保護者**: 毎回ブラウザでURLを開くのが面倒。通知が来ない。ホーム画面にアプリがない |
| **解決** | PWA（manifest.json + Service Worker）で「ホーム画面に追加」を実現 |
| **マネタイズ** | 一般公開の前提条件。将来的にApp Store掲載 → 保護者向け月額サブスクの導線確保 |
| **実装規模** | 小（PWA: 1-2日）。ただし一般公開基盤と同時実施のため全体では中 |

##### モバイル化の進め方

**結論: Webアプリの価値を先に証明してから、モバイル化と同時に一般公開基盤を整備する**

現状から並行してネイティブアプリを開発するのではなく、まずWebの機能を完成させる。

理由:
1. **今の最大の課題は「機能の価値」であって「配布経路」ではない** — 演習問題集入力や認証本番化など、Webでもモバイルでも共通で必要な機能が未完成
2. **PWAで十分な段階** — ターゲットユーザーはiPad/スマホでブラウザアクセス中。PWAで「アプリっぽい体験」は十分提供できる
3. **ネイティブが必要になるのはユーザーが増えた段階** — App Store掲載は塾外ユーザーにリーチしたいとき
4. **移行パスは確保されている** — Capacitor（WebViewベース）でNext.jsコードをほぼそのままiOS/Androidアプリ化可能

```
今          → 演習問題集で生徒価値を証明（Phase 1）
次          → 認証本番化 + PWA化 + 一般公開基盤整備（Phase 2）
ユーザー拡大時 → Capacitorでネイティブ化（必要に応じて）
```

##### 一般公開時に同時整備すべき基盤

モバイル化（= 不特定多数に開く判断）のタイミングで、以下を同時に整備する。
塾内利用の段階ではこれらは不要（対面説明・手動対応で代替可能）。

| 基盤 | なぜ一般公開時に必要か | 塾内利用では不要な理由 |
|------|----------------------|---------------------|
| **決済基盤（Stripe）** | 不特定多数から自動課金する手段がないとマネタイズ不可 | 請求書・銀行振込で対応可 |
| **利用規約・プライバシーポリシー** | 児童の学習データを扱うアプリで規約なしの公開は法的リスク | 保護者に対面で説明・同意を得られる |
| **APIレート制限** | AI系エンドポイントが無制限に叩かれるとAPI費用が爆発する | ユーザーが限定的で悪用リスクが低い |
| **保護者向け自動通知・進捗レポート** | 対面で会えない保護者にログインなしで価値を届ける必要がある | 塾の面談やLINEで直接伝えられる |
| **プロダクトアナリティクス** | ユーザー行動データなしでは改善の優先度が感覚頼みになる | ユーザーに直接ヒアリングできる |

##### 技術選定（Phase 3着手時に検討）

現時点では技術未確定。iOS標準の音声入力は品質がいまいちなため、以下の選択肢を比較検討する。

| 選択肢 | 概要 | メリット | デメリット |
|--------|------|---------|-----------|
| Web Speech API | ブラウザ内蔵 | コストゼロ、実装が簡単 | iOS Safariの認識精度が低い。日本語の変換品質に課題 |
| OpenAI Whisper API | ChatGPT/Typeless/Aqua Voiceの元技術 | 高精度。日本語対応も良好 | API呼び出しコストが発生（$0.006/分）。音声データの送信が必要 |
| Whisperセルフホスト | Whisperモデルを自前サーバーで実行 | API費用なし。カスタマイズ可能 | サーバー運用コスト。GPUインスタンスが必要 |
| Typeless / Aqua Voice等のAPI | 高品質な音声入力サービスのAPI利用 | Whisperベースで高品質。SDK提供あり | サービス依存。コスト体系の確認が必要 |

**現時点の所感**: Typeless/Aqua Voiceの品質が良いことは確認済み。これらの元技術であるOpenAI Whisper APIを直接使うか、サービスAPIを利用するかが主な比較軸。Phase 3着手時の技術動向・価格を見て判断する。

## 推奨実装ロードマップ

#### Phase 1 の設計詳細: 既存テーブル拡張方式（Codexレビュー反映）

**重要: 新テーブル（exercise_questions / exercise_answers）は作成しない。**
既存の`question_sets` / `questions` / `answer_sessions` / `student_answers`を拡張する。

```sql
-- 既存テーブルへのカラム追加

-- question_sets: 問題セットの種別を区別
ALTER TABLE question_sets ADD COLUMN set_type VARCHAR(20)
  NOT NULL DEFAULT 'master_print'
  CHECK (set_type IN ('master_print', 'exercise_workbook'));

-- questions: コース別フィルタ用
ALTER TABLE questions ADD COLUMN min_course course_level;
-- 既存のmaster_print問題はNULL（全コース表示）
-- exercise_workbook問題はmin_courseを設定

-- コース比較用の安全な関数（ENUM定義順に依存しない）
CREATE OR REPLACE FUNCTION course_rank(c course_level)
RETURNS SMALLINT IMMUTABLE LANGUAGE SQL AS $$
  SELECT CASE c
    WHEN 'A' THEN 1
    WHEN 'B' THEN 2
    WHEN 'C' THEN 3
    WHEN 'S' THEN 4
  END;
$$;
```

到達度マップ表示時のクエリ:
```sql
-- 現在のコースがBの場合
SELECT q.*, sa.is_correct
FROM questions q
JOIN question_sets qs ON q.question_set_id = qs.id
LEFT JOIN student_answers sa ON q.id = sa.question_id
  AND sa.answer_session_id IN (
    SELECT id FROM answer_sessions
    WHERE student_id = ? AND is_latest = true
  )
WHERE qs.set_type = 'exercise_workbook'
  AND course_rank(q.min_course) <= course_rank('B'::course_level)
  AND qs.grade = ?
  AND qs.subject_id = ?
ORDER BY qs.session_id, q.display_order;
```

**マルチアテンプト対応**: 既存の`answer_sessions`の`attempt_number` + `is_latest`設計をそのまま活用。
生徒が再挑戦した場合、新しい`answer_session`が作られ、前回の`is_latest`がfalseになる。
到達度マップは常に`is_latest=true`の回答を参照。過去の回答履歴も保持される。

## 設計判断の記録

### スキーマ設計: なぜ既存テーブル拡張か（新テーブルではなく）

既存DB（`20260211000001_create_auto_grading_tables.sql`）に以下のテーブルが存在する:
- `question_sets` — 問題セットコンテナ（セッション×科目×表示順）
- `questions` — 個別問題（numeric/fraction/multi_part/selection型対応、正解・配点付き）
- `answer_sessions` — 生徒の回答セッション（`attempt_number` + `is_latest`でマルチアテンプト対応）
- `student_answers` — 個別回答（3状態: true/false/NULL）

新テーブルを作ると:
1. **責務が重複** — 問題マスタ、回答記録、採点ロジック、RLSポリシーが二重管理になる
2. **マルチアテンプト非対応** — `UNIQUE(student_id, question_id)`では再挑戦履歴が消える（既存は`answer_sessions`で管理）
3. **既存ロジックの再利用不可** — `lib/math-grading.ts`、RPCsuch as `get_math_grading_history()`、RLSポリシーが使えない

拡張方式なら:
- `question_sets.set_type`で`master_print` / `exercise_workbook`を区別
- `questions.min_course`でコース別フィルタを追加
- 既存のマルチアテンプト、RLS、採点ロジックをそのまま活用

### コース比較: なぜ`course_rank()`関数を使うか

`min_course <= 'B'`のようなENUM直接比較は、PostgreSQLのENUM定義順（`CREATE TYPE course_level AS ENUM ('A','B','C','S')`）に依存する。
将来値の追加（例: `'A+'`）で比較結果が壊れるリスクがある。

`course_rank(course_level) RETURNS SMALLINT`関数でA=1, B=2, C=3, S=4と明示変換することで:
- ENUM定義順に依存しない安全な比較
- 将来の値追加時も関数を更新するだけ
- `IMMUTABLE`なのでインデックスにも使える

### コース別到達度: なぜ min_course 方式を採用するか

1. **問題IDがコース横断で共通** — コース変更してもデータが分断されない
2. **表示フィルタのみで切替** — `WHERE course_rank(min_course) <= course_rank(現在のコース)` だけで実現。追加テーブル不要
3. **コース変更で分母が変わる** — コースが上がると対象問題数が増え、到達率が相対的に下がり「伸びしろ」が可視化される
4. **コース別プレビューが容易** — 実際のコース設定を変えずに、マップの表示だけを別コース視点に切り替えられる
5. **セルフコンパッション** — Aコースの子は基本問題だけで100%を達成でき、成功体験を得やすい

### コース別プレビュー: なぜコース変更と分離するか

実際のコース設定（`students.course`）を変更すると、Spark画面の入力対象も変わってしまう。
到達度の「確認」と入力の「設定」を分離することで、安全に異なるコース視点を確認できる。
UIでは「プレビュー表示」ラベルを明示し、実コース指標との混同を防ぐ。

### 予習シリーズ廃止: なぜ段階的か

1. 一気に削除すると「今まで入れたデータが消えた」という不信感
2. 利用状況をモニタリングし、実際に入力が減ったことを確認してから非表示にする
3. 過去データは閲覧可能に残すことで、移行期のリスクを最小化
4. AI分析の入力源を`study_logs`から演習問題集データに段階移行する必要がある

### 一般公開基盤: なぜ生徒価値の証明後か

1. **塾内利用の段階では不要** — 決済は請求書、規約は対面説明、通知はLINE、分析は直接ヒアリングで代替できる
2. **モバイル化 = 「不特定多数に開く」判断** — このタイミングで初めて法的基盤（規約）、技術的基盤（レート制限）、収益基盤（Stripe）、リテンション基盤（通知・レポート）が必要になる
3. **生徒の価値体験が先** — 基盤を先に整えても、肝心のプロダクト価値がなければ意味がない。Phase 1で「成績が上がる」を証明してからPhase 2で公開基盤を整える

### Phase順序: なぜ演習問題集が認証より先か

1. **現在の塾内ではデモ認証でも運用できている** — 認証本番化は「一般公開」に必要であって「塾内利用」には緊急ではない
2. **演習問題集こそがプロダクトのコア価値** — 認証を本番化しても、肝心の機能がなければ有料化の根拠がない
3. **価値が証明できてから有料化ステップへ** — Phase 1で「成績が上がる」を実証し、Phase 2で認証・課金・公開基盤を一気に整備する流れが自然

### Phase 2ゲート方式: なぜ法務→認証→課金の順か

1. **法務が最初**: 利用規約・PP・特商法表示・保護者同意記録がないとStripe導入しても法的に課金開始できない
2. **認証が次**: 本番認証が動かないとユーザー管理ができず、課金対象の特定ができない
3. **課金が最後**: 法務と認証が整った上で初めて安全に課金を開始できる
4. **参考法令**: 個人情報保護法（児童データの法定代理人同意）、特定商取引法（通信販売の表示義務）、民法（未成年者契約）
